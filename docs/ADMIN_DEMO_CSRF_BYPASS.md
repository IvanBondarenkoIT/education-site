# Вход в админку на Railway: CSRF и временный обход

## Почему бывает 403 при входе в `/admin/`

1. **Прокси и HTTPS** — Gunicorn за Railway не видит TLS; без `SECURE_PROXY_SSL_HEADER` Django считает запрос HTTP, а браузер шлёт `Origin: https://...` — расхождение.
2. **`CSRF_TRUSTED_ORIGINS`** — домен деплоя должен попадать в список (включая `https://*.railway.app` и т.д.).
3. **Ветка Git** — Railway часто деплоит **`main`**, а фиксы лежат в **`django-development`**: на сервере старый код → старые настройки CSRF.
4. **Куки / SameSite** — реже, но бывает при смешанном http/https или блокировке сторонних кук.

У других на Railway с Django та же связка: **X-Forwarded-Proto + CSRF_TRUSTED_ORIGINS + правильная ветка**. Поиск: *Django Railway CSRF*, *Origin checking failed railway.app*.

## План действий (в порядке приоритета)

| Шаг | Действие | Зачем |
|-----|----------|--------|
| 1 | В Railway: **какая ветка** деплоится? Совпадает ли с веткой, куда вы пушите? | Иначе фиксы не доезжают |
| 2 | Убедиться, что на сервере последний `settings.py` (proxy + CSRF origins) | Устраняет типичную причину 403 |
| 3 | В Variables: `DJANGO_CSRF_TRUSTED_ORIGINS=https://ВАШ-ТОЧНЫЙ-URL.railway.app` | Запасной вариант под ваш домен |
| 4 | **Демо пользователю без правки CSRF:** локально `runserver` + **screen share** или **туннель** (ngrok) | Нулевой риск для продакшена |
| 5 | **Временный обход** (ниже): только если 1–3 не помогли и нужен срочный показ на том же URL Railway | Явный риск, сразу выключить после демо |

## Временный обход: `ADMIN_DEMO_BYPASS_CSRF`

В коде включён middleware `AdminDemoBypassCsrfMiddleware` **перед** `CsrfViewMiddleware`.

Он **ничего не делает**, пока в окружении **нет** переменной.

### Включить (только на время демо)

В **Railway → Variables** добавить:

```text
ADMIN_DEMO_BYPASS_CSRF=1
```

Сделать **Redeploy**. После этого POST на `/admin/login/` проходит без проверки CSRF (включая проверку Origin).

### Обязательно после демо

1. **Удалить** переменную `ADMIN_DEMO_BYPASS_CSRF` из Railway.  
2. При желании убрать строку middleware из `settings.py` (можно оставить — без переменной он пассивен).

### Риски

- Пока флаг **включён**, теоретически проще атаки вида login CSRF (злоумышленник заставляет браузер жертвы отправить форму логина с известными кредами — сценарий редкий, но для публичного URL это ослабление защиты).
- Используйте **сильный пароль** админа и **отключите флаг** сразу после показа.

---

## Альтернатива без отключения CSRF

- Показать админку с **локального** `python manage.py runserver` (данные можно подтянуть через dump БД или тестовые фикстуры).
- Или одноразово смержить `django-development` → `main` и задеплоить, чтобы заработали постоянные CSRF-фиксы.
